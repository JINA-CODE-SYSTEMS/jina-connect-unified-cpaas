The voice E2E suite (`voice/tests/test_e2e.py`) hits real provider
sandboxes — actual outbound calls to Twilio / Plivo / Vonage / Telnyx
/ Exotel test numbers, and optionally a SIP trunk via Asterisk. It is
gated by `VOICE_E2E_TESTS=1` so it never runs on a regular PR by
accident, and uses sandbox credentials to keep cost negligible.

## When it runs

The `voice-e2e.yml` workflow runs in three situations:

  1. **Weekly schedule** — Sundays 03:00 UTC. Catches silent
     breakages in any sandbox before a release.
  2. **Manual `workflow_dispatch`** — kick off a run from the GitHub
     Actions tab, optionally toggling the SIP sub-suite.
  3. **`e2e:voice` label on a PR** — maintainers add the label when
     they want a PR to exercise the real providers before merging.

Regular PR CI never triggers it, so the main pipeline stays fast.

## Required GitHub Secrets

Each provider's tests skip if any of its required env vars is missing.
Set whichever you want covered.

<Aside type="note">
  All values here are stored as GitHub repository secrets (Settings →
  Secrets and variables → Actions). The workflow reads them via
  ``${{ secrets.X }}``; nothing is logged.
</Aside>

### Twilio

| Secret                       | Notes                                                            |
|------------------------------|------------------------------------------------------------------|
| `VOICE_E2E_TWILIO_SID`       | Account SID (starts with `AC...`).                               |
| `VOICE_E2E_TWILIO_TOKEN`     | Auth token. Use the test credentials, not production.            |
| `VOICE_E2E_TWILIO_FROM`      | E.164 outbound number bought on the sandbox account.             |
| `VOICE_E2E_TWILIO_TO`        | Twilio test-recipient (`+15005550006`) or your own verified DID. |

## Running locally

```bash
export VOICE_E2E_TESTS=1
# Set as many of the per-provider env vars as you want to cover —
# tests skip individually if their creds are missing.
export VOICE_E2E_TWILIO_SID=ACxxxx
export VOICE_E2E_TWILIO_TOKEN=...
export VOICE_E2E_TWILIO_FROM=+15005550006
export VOICE_E2E_TWILIO_TO=+15005550006

pytest voice/tests/test_e2e.py -v
```

The suite uses `TransactionTestCase` so each test's DB writes are
visible to the Celery worker handling provider webhooks (the regular
`TestCase` would roll the txn back and hide them).

## What each test asserts

| Test                                            | Asserts                                                                                                  |
|-------------------------------------------------|----------------------------------------------------------------------------------------------------------|
| `test_outbound_call_lifecycle`                  | Adapter replaces the placeholder `provider_call_id`, call reaches a terminal state inside 90s, `VoiceCallEvent` rows are present in monotonic sequence order. |
| `test_recording`                                | `VoiceRecording` row lands with a `storage_url`, `storage.signed_url(...)` returns an `http(s)://` URL.  |
| `test_transcription`                            | After recording, polls `VoiceRecording.transcription` for up to 90s and asserts a non-empty string.       |
| `TestInboundE2E.test_inbound_call_creates_...`  | A `VoiceCall(direction=inbound)` row appears for the configured provider within 90s, plus a `team_inbox.Messages` row from the post-call signal. |
| `TestSIPE2E.test_sip_outbound_call_reaches_...` | Same as the HTTP outbound lifecycle but routed through Asterisk.                                          |

## Diagnosing a red run

The workflow uploads `voice-e2e.log` as an artifact (`voice-e2e-log`),
retained for 14 days. The pytest output captures each test's poll
loop, the last observed `VoiceCall.status`, and the provider's most
recent webhook payload (via `voice.tasks.process_call_status` info
logs).

When debugging locally, run with `-s` to stream logs without buffering
and tail `voice-e2e.log` as the test runs:

```bash
pytest voice/tests/test_e2e.py::TestTwilioE2E -v -s
```
